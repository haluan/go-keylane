# v0.3 Request Runtime Testing

This document describes the test coverage for the v0.3 request runtime. All tests are in the `httpkeylane` package using standard Go tooling.

---

## Overview

The v0.3 test suite proves that the complete HTTP request execution path works correctly from `net/http` middleware through the Keylane scheduler:

```text
HTTP request
  -> key extraction
  -> lane mapping
  -> request metadata creation
  -> admission control
  -> SubmitRequest[I, O]
  -> shard routing
  -> lane routing
  -> scheduler execution
  -> Future completion
  -> HTTP response
```

Tests use `net/http/httptest` for HTTP integration and `context` cancellation/deadline for runtime behavior. No external test dependencies are required.

## Test Categories

### 1. Basic Middleware Execution

- Handler runs through the Keylane runtime (not directly).
- Middleware waits for scheduled handler completion before returning.
- Handler does not run before worker executes it.

### 2. Key Extraction Integration

- `HeaderKey` routes requests by header value.
- `QueryKey` routes requests by query parameter.
- `FirstNonEmptyKey` falls back to secondary key sources.
- `CompositeKey` produces stable routing keys from multiple parts.
- Missing key returns 400 and does not enqueue.

### 3. Lane Mapping Integration

- `MethodLaneMapper` maps GET to read lane, POST to write lane.
- `RouteLaneMapper` applies the first matching rule.
- More specific rules placed first take priority.
- Fallback lane mapper applies when no rule matches.
- Invalid lane returns 400 and does not enqueue.

### 4. Request Runtime Routing

- Same key always routes to the same shard.
- Different keys may route to different shards.
- Configured lane controls which lane queue receives the request.
- Queue wait duration is recorded after the worker starts.
- Run duration is recorded after the handler returns.

### 5. Cancellation and Timeout

- Request cancelled before execution: handler does not run.
- Deadline expires while queued: handler does not run.
- Cancellation while running: handler observes `ctx.Done()`.
- Handler ignoring cancellation: runtime does not panic; middleware returns safely.
- Await timeout (via context cancellation): no goroutine leak.

### 6. Admission Rejection

- Admission disabled: requests admitted under high queue pressure.
- Admission enabled: requests rejected before enqueue when pressure ≥ threshold.
- Default rejection status: 503.
- Status override: 429 when `RejectStatusCode: 429`.
- Custom `ErrorHandler`: called with `ErrAdmissionRejected`.
- Rejection counter: `AdmissionRejected` lane counter increments; `TotalQueued` does not.

### 7. Queue Full and Submit Failure

- Queue full: handler does not run; middleware returns 503.
- Stopped queue: middleware returns error response without panic.
- Custom `ErrorHandler`: called with submit failure error.

### 8. Observability Metadata

- `X-Request-ID` header propagated to `RequestObservation.RequestID`.
- `Transport` is set to `"http"` for all HTTP middleware requests.
- `OperationFunc` sets `Operation` on the observation.
- `Key`, `Lane`, and `ShardID` are present in completed observations.
- Outcome classification: `completed`, `cancelled`, `timed_out`, `admission_rejected`, `rejected`.

### 9. HTTP Status Capture

- Handler writing body without `WriteHeader` → status 200.
- Handler calling `WriteHeader(201)` → status 201.
- Middleware error (missing key) → status 400.
- Multiple `WriteHeader` calls → first write wins.

### 10. Concurrent Requests

- N concurrent requests complete without races or panics.
- Same-key requests always route to the same shard under concurrency.
- Mixed GET/POST requests update read and write lane counters.
- Overload with admission enabled: some reject safely; no deadlock or panic.

### 11. Goroutine Leaks

- Queued cancellation: goroutine count returns to baseline after drain.
- Admission rejection: goroutine count returns to baseline after drain.
- Queue full rejection: goroutine count returns to baseline.
- Concurrent overload: goroutine count returns to baseline after drain.

## Test Helpers

### `controlledHandler`

Blocks and releases handler execution deterministically:

```go
type controlledHandler struct {
    started chan struct{}
    release chan struct{}
    ran     atomic.Bool
}
```

Use `<-ch.started` to wait for handler entry, then `close(ch.release)` to unblock it.

### `testRequestObserver`

Thread-safe observer with wait methods:

```go
type testRequestObserver struct {
    mu        sync.Mutex
    queued    []keylane.RequestMeta
    started   []keylane.RequestObservation
    completed []keylane.RequestObservation
    rejected  []keylane.RequestObservation
}
```

Methods: `waitStarted(t, n)`, `waitCompleted(t, n)`, `waitRejected(t, n)` — poll with a 3-second deadline.

### `eventuallyNoGoroutineGrowth`

Polls `runtime.NumGoroutine()` until count returns near the baseline or a deadline expires:

```go
func eventuallyNoGoroutineGrowth(t *testing.T, before int, tolerance int)
```

Use this after stopping queues and waiting for drain to confirm no goroutine leak.

### `newTestQueue`

Creates and starts a queue with cleanup and drain on `t.Cleanup`:

```go
func newTestQueue(t *testing.T, opts ...testQueueOption) (*keylane.Queue, context.Context)
```

Options: `withShardCount`, `withWorkerCount`, `withQueueSizePerLane`, `withLaneQuotas`, `withRequestHooks`, `withObservability`.

## Determinism Notes

The test suite avoids fragile sleep-based synchronization. Preferred patterns:

- Channels for blocking and releasing handler execution.
- `atomic.Bool` for handler-ran assertions.
- Observer polling with explicit timeout (3 seconds default).
- `blockWorker` to hold a worker so requests remain queued.
- `waitDone` helper around `time.After` to detect deadlocks.

Short `time.Sleep` calls are used only where deterministic synchronization is not practical (e.g., waiting for queue pressure to stabilize).
