# Migration: v0.7 ŌåÆ v0.8

v0.8 is a **pre-v1.0** hardening release. Most callers require no code changes. This note covers KL-1801 through KL-1807 (API stability, config, defaults, observability, benchmarks, lifecycle, examples).

---

## No action required

If you use only:

- `Queue`, `New`, `Submit`, `SubmitValue`, `SubmitRequest`
- v0.6 failure/retry/deadline configuration
- `httpkeylane.Middleware`

ŌĆ”you do not need to migrate for v0.8 unless you relied on symbols listed below.

---

## New: configuration validation (KL-1802)

Optional but recommended before production:

- `ValidateConfig(cfg)` ŌĆö structured errors and warnings with stable codes (`KL_CONFIG_*`)
- `NormalizeConfig(cfg)` ŌĆö redacted snapshot of effective settings
- `Queue.ConfigValidationWarnings()` ŌĆö warnings from construction

**No breaking change** for valid configs that already passed `New`. New fatal rule:

| Rule | Impact |
|------|--------|
| `Retry.MaxAttempts` > 256 when retry enabled | `New` returns `ErrInvalidRetryPolicy` |

See [config-validation.md](../config-validation.md), [config-versioning.md](../config-versioning.md), and [production-defaults.md](../production-defaults.md).

## New: production safety defaults (KL-1803)

Recommended starting configuration:

```go
cfg := keylane.ProductionDefaults()
report := keylane.ExplainDefaults(cfg)
```

| Addition | Notes |
|----------|--------|
| `ProductionDefaults()` | Bounded core scheduler; retry/continuation/backend disabled; low-allocation observability |
| `ExplainDefaults(cfg)` | Inspects subsystem gates and validation warnings |
| `KL_CONFIG_OBSERVABILITY_FULL_DEFAULTS_RESOLVED` | Warning when unset `Observability` resolves to full defaults at `New` |
| `KL_CONFIG_DEBUG_SNAPSHOT_HOT_PATH_HEAVY` | Warning when debug snapshot + worker timing are all enabled without low-allocation mode |

**No breaking change** for valid v0.7 configs that already passed `New`. Risky subsystems remain opt-in.

See [production-defaults.md](../production-defaults.md), [compatibility-rules.md](../compatibility-rules.md), and [observability-contract.md](../observability-contract.md).

## New: observability contract (KL-1804)

| Addition | Notes |
|----------|--------|
| `StableMetricDescriptors()` | Canonical stable prometheus metric inventory (no v0.7 renames) |
| `ExperimentalMetricPatterns()` | Hook-adapter metrics (pipeline, continuation, backend) |
| `ForbiddenMetricLabelNames()` | Labels that must not appear on stable metrics |
| `otel.StableTraceAttributeKeys()` | Documented OTEL span attributes |
| Contract tests | Prometheus gather + label policy locked in CI |

**No breaking change** for prometheus metric names or labels from v0.7. `DebugSnapshotVersion` remains `"6"` unless release notes say otherwise.

See [observability-contract.md](../observability-contract.md), [production-hardening.md](../production-hardening.md), and [releases/v0.8.0.md](../releases/v0.8.0.md).

Hook payloads redact raw `Key` and `RequestID` by default. Opt in with `Observability.ExposeRawRequestIdentifiers` only when required (validation warns with `KL_CONFIG_RAW_REQUEST_IDENTIFIERS_IN_HOOKS`).

## Migration checklist

1. Update dependency to v0.8.
2. Run compile checks.
3. Replace renamed APIs (`AdaptiveQuotaSnapshot` ŌåÆ `AdaptiveDebugSnapshot`).
4. Prefer stable APIs over experimental where possible.
5. Run `ValidateConfig` / review `ExplainDefaults`.
6. Review default behavior (`ProductionDefaults` vs hand-rolled config).
7. Review observability names, labels, and hook redaction defaults.
8. Run unit tests.
9. Run `go test -race ./...` (and adapter modules).
10. Run representative benchmarks (`make bench-production` or `make bench-baseline`).
