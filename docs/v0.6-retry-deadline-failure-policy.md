# v0.6.0 ‚Äî Retry, Deadline & Failure Policy

Milestone **v0.6** makes failure handling explicit inside `go-keylane`. Retries are not automatic for every error ‚Äî they are **classified**, **bounded**, **deadline-aware**, **duplicate-safe**, **pressure-aware**, and **observable**.

> **Keylane does not make retries automatic for every error.**  
> It makes retries bounded, deadline-aware, idempotency-aware, and pressure-aware.

Keylane is **not** a circuit breaker, workflow engine, or distributed retry coordinator. Retry runs **in-worker** on the same process; it does not coordinate retries across replicas or persist retry state across restarts.

## End-to-end decision pipeline

```text
SubmitValue / SubmitRequest
  ‚Üì
context deadline captured as DeadlineBudget
  ‚Üì
handler attempt
  ‚Üì
FailurePolicy classifies error ‚Üí FailureKind
  ‚Üì
RetryPolicy: retry eligible? attempts left? backoff? budget for delay?
  ‚Üì
IdempotencyPolicy: duplicate-safe?
  ‚Üì
RetrySuppressionPolicy: pressure, lane, shard, hot key, scale signal?
  ‚Üì
retry scheduled OR stopped OR suppressed
  ‚Üì
Future trace + RetryFailureSnapshot + optional hooks
```

Three gates run **before** a retry sleep (when retry is enabled):

1. **DecideRetry** ‚Äî failure kind, max attempts, context, deadline budget  
2. **DecideRetrySafety** ‚Äî idempotency metadata and hooks  
3. **DecideRetrySuppression** ‚Äî runtime health (pressure, overload, hot key, etc.)

Suppression does not change the handler error returned to the caller; it is recorded on `RetryTrace`.

## Safe defaults

| Setting | Default behavior |
|---------|------------------|
| `RetryPolicy` zero value | Retry **disabled** |
| `RetrySuppressionPolicy` zero value | Suppression **disabled** |
| `Idempotency.Safety` unspecified + retry on | Treated as **unsafe** ‚Äî no silent retries |
| Unknown handler errors | **Not** retryable unless wrapped or classified |

Enable retry, idempotency policy, and suppression deliberately per environment and lane class.

## Quick examples

### Minimal retry (read-only)

```go
future, err := keylane.SubmitValue(ctx, q, keylane.ValueJob[string]{
    Key:  "user-123",
    Lane: "read",
    Retry: keylane.RetryPolicy{Enabled: true, MaxAttempts: 3},
    Idempotency: keylane.Idempotency{Safety: keylane.RetrySafetySafe},
    Run: func(ctx context.Context) (string, error) {
        return fetchProfile(ctx)
    },
})
```

### Unsafe mutation (no automatic retry)

```go
future, err := keylane.SubmitValue(ctx, q, keylane.ValueJob[string]{
    Key:  "payment-123",
    Lane: "payment",
    Retry: keylane.RetryPolicy{Enabled: true, MaxAttempts: 3},
    Idempotency: keylane.Idempotency{Safety: keylane.RetrySafetyUnsafe},
    Run: chargePayment,
})
// Retryable handler errors will not schedule another attempt unless
// AllowUnsafeRetry or an idempotency hook explicitly allows it.
```

### Retry trace

```go
trace, ok := keylane.RetryTraceFromFuture(future)
if ok {
    fmt.Println(trace.Final.Succeeded)
    fmt.Println(trace.Final.SuppressionReason)
    fmt.Println(trace.Final.StoppedReason)
}
```

### Failure snapshot (pull, may allocate)

```go
snap := q.RetryFailureSnapshot()
fmt.Println(snap.RetriesScheduledTotal)
fmt.Println(snap.RetriesSuppressedTotal)
```

## Operational trade-offs

| Choice | Benefit | Risk |
|--------|---------|------|
| Higher `MaxAttempts` | More tolerance for transient errors | Longer worker occupancy, client wait |
| Aggressive backoff | Spreads retry storms | May exceed caller deadline |
| `RetrySafetySafe` on writes | Convenient retries | Duplicate side effects if business logic is not idempotent |
| Suppression on | Protects queue under pressure | Fewer internal retries; clients may still retry |
| Hooks enabled | Rich debugging | Extra allocation and callback cost |

---

## Common mistakes

- Enabling retry for **unsafe mutations** without idempotency keys or hooks  
- Retrying after the **caller deadline is already exhausted** (check `BudgetFromFuture`)  
- Using **raw keys, request IDs, or idempotency keys** as metric labels  
- Treating **retry suppression** as a successful handler outcome ‚Äî the original failure still returns  
- Retrying **overload** or **admission** failures by default ‚Äî amplifies load  
- Expecting Keylane to provide **durable exactly-once** or **cross-replica** retry coordination  
- Confusing Keylane with a **circuit breaker** or **workflow engine**

---

## Required warnings

- Retry is **disabled by default** (zero `RetryPolicy`).  
- Retry is bounded by **`MaxAttempts`**.  
- Retry is bounded by **caller deadline budget** (`MinRemainingBudget`).  
- Retry is **blocked for unsafe jobs** by default when retry is enabled.  
- Retry may be **suppressed under pressure**.  
- Retry is **not** a circuit breaker.  
- Retry is **not** a workflow engine.  
- Retry does **not** make side effects idempotent.  
- Retry observability must avoid **high-cardinality** labels.

## Migration from v0.5

v0.5 focused on hot keys, per-key admission, shard pressure, and scale signals. v0.6.0 adds an **opt-in** retry stack on top:

1. Add `Config.Retry` only where in-process retry is appropriate.  
2. Set `Idempotency` on every retried job; use `Config.Idempotency` for cluster-wide `RequireForRetry`.  
3. Enable `Config.RetrySuppression` in production to limit retry storms when queues are pressured (complements v0.5 admission).  
4. Use `FailureFromFuture` / `RetryTraceFromFuture` instead of inferring behavior from plain `error`.  
5. Pull `RetryFailureSnapshot()` on scrape intervals ‚Äî not on every request path.  
6. Map lane classes (`critical`, `best_effort`, `background`) via [admission policy](admission-control.md) for suppression behavior.

Existing v0.5 metrics and `DebugSnapshot` fields are unchanged; v0.6.0 adds retry/failure counters and traces.
