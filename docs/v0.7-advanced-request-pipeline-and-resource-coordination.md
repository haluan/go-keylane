# v0.7.0 — Advanced Request Pipeline & Backend Resource Coordination

Milestone **v0.7** extends the v0.6 request runtime with multi-stage pipelines, stage-aware execution context, non-blocking continuations, in-process backend resource coordination, optional DB/API pool pressure adapters, and consolidated observability guidance.

> **Keylane v0.7.0 is in-process request orchestration.** It is not a persistent workflow engine, distributed queue, database transaction manager, or exactly-once execution system. Public APIs may still evolve before v1.0. See [api-stability.md](api-stability.md) and [public-api-inventory.md](public-api-inventory.md).

Built on [v0.6 retry, deadline, and failure policy](v0.6-retry-deadline-failure-policy.md) and [v0.5 hot-key / autoscaling signals](v0.5-hot-key-autoscaling-signals.md).

---

## Why v0.7.0 exists

v0.6 made failures **classified**, **bounded**, and **observable** for single-handler requests (`SubmitRequest`, `SubmitValue`). Production handlers are often multi-step: validate, authorize, database reads, external APIs, and response assembly. A single handler hides where time was spent, which step failed, and how downstream pools relate to queue pressure.

v0.7.0 adds:

| Feature | What it adds |
|---------|----------------|
| Pipeline stages | Ordered `SubmitPipeline` stages with shared state |
| Stage context | `StageExecutionContext` on `context.Context` |
| Continuations | Non-blocking continuations (yield/resume on same key shard) |
| Backend coordination | In-process backend resource/lane admission (`AcquireBackend`, `WithBackend`) |
| Pool pressure adapters | Optional DB/API pool pressure adapters (observational) |
| Observability & testing | Cross-feature tests, benchmarks, hook/metric guidance |

v0.6 semantics are preserved: admission, overload, per-key admission, retry, idempotency, suppression, deadline budget, `Future`, and failure classification all apply to pipelines the same way as to `SubmitRequest`.

---

## SubmitPipeline vs SubmitRequest

| Use `SubmitRequest` | Use `SubmitPipeline` |
|---------------------|----------------------|
| Single logical step | Multiple ordered steps with shared state |
| No stage-level metrics needed | Stage hooks, duration, and failure attribution |
| No continuation yield | Slow I/O outside the worker via continuations |
| No per-stage backend lanes | Bounded downstream usage per resource/lane |

Existing `SubmitRequest` callers **do not need to migrate**. Pipelines are opt-in.

## How the pieces fit together

```text
SubmitPipeline
  → same Key/Lane routing and admission as SubmitRequest
  → stage[0] … stage[n] (sync Run or RunContinuation yield)
  → optional WithBackend per stage
  → optional BackendPressure probes (observational)
  → Complete → Future resolves once
```

1. **Stages** — Sequential in-worker execution; first error stops the pipeline; `Complete` runs only after all stages succeed.
2. **Stage context** — Each stage reads `StageExecutionFromContext` for shard, lane, stage name, attempt, queue wait, and deadline snapshot.
3. **Continuations** — A stage can yield: worker released, async work calls `ContinuationCompleter.Complete`, resume job runs on the **same key shard**. Shard identity end-to-end: **yes**. Worker blocked end-to-end: **no**.
4. **Backend coordination** — `WithBackend` admits a lease against configured `MaxInFlight` per resource/lane; saturated lanes reject when admission mode is `reject`.
5. **Pool pressure** — Adapters map `database/sql` or custom API semaphores into `BackendPressureSnapshot`. Keylane does **not** auto-reject from pool telemetry unless your application gates on the snapshot.
6. **Observability** — Request, stage, continuation, backend, and pressure hooks; `DebugSnapshot` v0.7.0 fields; metric naming in [metrics.md](metrics.md).

---

## Non-goals

v0.7.0 explicitly does **not** provide:

- Persistent workflows or saga orchestration
- Distributed queues or cross-replica execution guarantees
- Exactly-once delivery or automatic deduplication across restarts
- Database transaction management or two-phase commit
- Kubernetes or cloud autoscaler implementations (signals only)
- Per-stage retry policy (retry remains **request-level**)

Pipelines coordinate work **inside one process** on the existing lane-sharded runtime.
