# v0.7.0 Release Notes

`v0.7.0 — Advanced Request Pipeline & Backend Resource Coordination`

---

## Overview

v0.7.0 adds multi-stage request pipelines, stage-aware execution context, non-blocking continuations, in-process backend resource/lane coordination, optional DB/API pool pressure adapters, and consolidated observability guidance — on top of v0.6 retry, deadline, and failure policy.

Full guide: [v0.7.0 overview](../v0.7-advanced-request-pipeline-and-resource-coordination.md).

---

## Highlights

- **Request pipeline stage model** — `SubmitPipeline` with ordered `PipelineStage` and `StageFailure` attribution
- **Stage-aware execution context** — `StageExecutionFromContext`, attempt, deadline snapshot, queue wait
- **Non-blocking continuations** — yield/resume on same key shard; worker released during slow I/O
- **Backend resource lane coordination** — `AcquireBackend`, `WithBackend`, admission and `DebugSnapshot.BackendResources`
- **DB/API pool pressure adapters** — `SQLDBPressureAdapter`, `APIClientPressureAdapter`, observational snapshots
- **Observability, tests, benchmarks** — hook lifecycle docs, test matrix, `BenchmarkPipeline*` suite

---

## Request pipeline stage model

- `SubmitPipeline[S,O]` with shared state type `S` and output type `O`
- Sequential in-worker stages; request-level retry restarts full pipeline
- Stage hooks: `OnStageStarted`, `OnStageCompleted`, `OnStageFailed`

See [request-pipeline.md](../request-pipeline.md).

---

## Stage-aware execution context

- `StageExecutionContext` on `context.Context` for every pipeline stage and `SubmitRequest` handler
- `StageMetaFromContext`, `RequestMetaFromExecution`, `BackendOperationFromStage`

See [stage-execution-context.md](../stage-execution-context.md).

---

## Non-blocking continuations

- Opt-in `ContinuationConfig.Enabled`
- `RunContinuation` + `NewContinuation` / `ContinuationCompleter`
- Same key/shard routing on resume; late completion accounting

See [continuations.md](../continuations.md).

---

## Backend resource lane coordination

- Per-resource, per-lane `MaxInFlight` with `BackendAdmissionReject`
- `BackendLease.Release` decrements in-flight counts
- Works in sync stages and continuation completer goroutines

See [backend-resource-coordination.md](../backend-resource-coordination.md).

## DB/API pool pressure adapters

- `BackendPressureProvider` snapshots for external pools
- `Queue.BackendPressure(ctx)` and `DebugSnapshot.BackendPressure`
- **Observational only** — keylane does not auto-reject from pool telemetry unless the application gates on snapshots

See [backend-pressure-adapters.md](../backend-pressure-adapters.md).

## Compatibility notes

- **No forced migration** — `SubmitRequest` and `SubmitValue` continue to work unchanged
- **Continuations are opt-in** — `Continuation.Enabled` defaults to off
- **Backend coordination is opt-in** — `BackendResources.Enabled` defaults to off
- **Pressure adapters are opt-in** — register `PressureProviders`; observational unless you gate calls
- **APIs may evolve before v1.0** — treat v0.7.0 as additive on v0.6

## Upgrade guidance

1. Read the [v0.7.0 overview](../v0.7-advanced-request-pipeline-and-resource-coordination.md).
2. Try `go run ./examples/pipeline_basics` and `./examples/pipeline_continuation`.
3. Enable features incrementally: pipelines first, then continuations, then `BackendResources`, then pressure providers.
4. Wire hooks per [pipeline-observability.md](../pipeline-observability.md); use low-cardinality labels from [metrics.md](../metrics.md).

---

## Non-goals

v0.7.0 does **not** provide:

- Persistent workflows or sagas
- Distributed queues or cross-replica exactly-once execution
- Database transaction management
- Built-in Kubernetes autoscaler or HPA integration (signals only)

Pipelines are **in-process request orchestration** on the existing lane-sharded runtime.

---

## Known limitations

- Retry is request-level only (no per-stage retry policy)
- `BackendAdmissionWait` queue mode is reserved, not implemented
- `net/http.Transport` has no universal pressure adapter — use custom bounded clients
- Continuation resolution does not cancel external I/O automatically
