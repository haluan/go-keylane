# v0.5.0 Release Notes

`v0.5.0 — Hot Key Mitigation, Shard Balancing & Autoscaling Signals`

---

## Overview

v0.5 adds production visibility for bounded concurrency systems: bounded hot key detection, per-key admission policy, shard pressure diagnostics, autoscaling signal metrics, and observability hooks/tests/benchmarks.

go-keylane helps operators distinguish normal backpressure from localized hot-key pressure, hot-shard pressure, hot-lane pressure, distributed backlog, and real scale-out demand — even when CPU and memory remain flat.

go-keylane does **not** replace your autoscaler. It exposes pressure signals that operators and platforms need when backpressure hides demand from CPU and memory metrics.

---

## Highlights

- **Bounded hot key detection** — fixed-size per-shard tracker, hash-only diagnostics by default
- **Per-key admission** — throttle/reject/shed noisy keys without punishing unrelated traffic
- **Shard pressure diagnostics** — classify localized vs distributed vs worker-bound pressure
- **Autoscaling signals** — advisory `ScaleSignal` with reason, scope, and sustained-window guardrails
- **v0.5 observability** — DebugSnapshot fields, hooks, Prometheus metrics, tests and benchmarks

## New APIs

### Hot key detection

- `HotKeyConfig`, `DefaultHotKeyConfig()`
- `DebugSnapshot().HotKeys`, `Shards[].HotKeyCandidate`, `HotKeyCandidates`

### Per-key admission

- `PerKeyAdmissionConfig`, `DefaultPerKeyAdmissionConfig()`
- `CheckPerKeyAdmission`, `PerKeyAdmissionError`
- `DebugSnapshot().PerKeyAdmissionSnapshots`, `Mitigations`

### Shard pressure

- `ShardPressureConfig`, `DefaultShardPressureConfig()`
- `Queue.PressureSummary()`, `Queue.ShardPressure()`, `Queue.HotShards()`
- `DebugSnapshot().PressureSummary`, `ShardPressure`

### Autoscaling signals

- `AutoscalingSignalConfig`, `DefaultAutoscalingSignalConfig()`
- `Queue.ScaleSignal()`, `ScaleSignal`, `ScaleReason`, `ScaleScope`
- `DebugSnapshot().ScaleSignal`

### Observability

- Hooks: `OnHotKeyCandidate`, `OnShardPressureSummary`, `OnScaleSignal` (plus existing `OnPerKeyAdmissionDecision`)
- Hook panic recovery on all emitters
- v0.5 Prometheus metrics — see [metrics.md](../metrics.md)

## Behavior changes

- `DebugSnapshot()` version `"4"` adds v0.5 top-level fields: `HotKeys`, `Mitigations`, `ShardPressure`
- Default configs enable v0.5 features when using `DefaultHotKeyConfig()` etc.; zero config structs disable each subsystem independently
- Scale signal requires sustained unhealthy windows before `Recommended=true`

No breaking changes to v0.3 request runtime or v0.4 adaptive quota/overload APIs.

---

## Migration notes

v0.5 is **additive**. Existing `Queue.Submit`, `SubmitValue`, and v0.3/v0.4 configuration continue to work.

To enable v0.5 features:

```go
cfg.HotKey = keylane.DefaultHotKeyConfig()
cfg.PerKeyAdmission = keylane.DefaultPerKeyAdmissionConfig()
cfg.ShardPressure = keylane.DefaultShardPressureConfig()
cfg.AutoscalingSignal = keylane.DefaultAutoscalingSignalConfig()
```

To stay on pre-v0.5 behavior, leave all four as zero values.

Observe-only hot keys: enable `HotKey`, disable `PerKeyAdmission` or use `PerKeyMitigationAllow`.

---

## Compatibility notes

- Core module has no new required dependencies
- Prometheus and OpenTelemetry remain optional adapter modules
- Raw key exposure disabled by default (`ExposeRawKey: false`)
- Per-key admission requires hot key tracking (`MaxTrackedKeysPerShard > 0`)

---

## Known limitations

- Scale signals are **advisory** — go-keylane does not calculate replica counts
- Hot key detection produces **candidates**, not confirmed root causes
- Bounded tracker counts are approximate; LRU eviction can miss ephemeral keys
- Raw key labels are disabled by default; do not add them to metrics
- Autoscaler wiring (HPA, KEDA, custom loops) is left to the consuming platform
- go-keylane does not eliminate Go GC pauses
