# v0.5 Hot Key, Shard Pressure & Autoscaling Signals

Milestone **v0.5.0** adds production visibility for bounded concurrency systems: hot key detection, per-key mitigation, shard pressure diagnostics, and autoscaling signal metrics.

> **CPU tells you how busy the process is.**  
> **Keylane tells you how much demand the process is holding back.**

> **Backpressure protects the process.**  
> **Autoscaling signals tell the platform when protection is no longer enough.**

go-keylane does **not** replace your autoscaler. It exposes pressure signals that operators and platforms need when backpressure hides demand from CPU and memory metrics.

---

## Why v0.5 matters

Bounded concurrency keeps CPU, memory, goroutine count, and GC pressure stable under load. That stability is good for survival, but it can confuse operators and autoscalers that only watch process CPU and memory.

A service can look calm while holding back demand through queueing, throttling, shedding, or rejecting work. v0.5 answers two production questions:

1. **Is the service overloaded globally** and likely needs scale-out?
2. **Or is overload localized** to one hot key, one lane, or one shard?

## Feature map

| Feature | Primary API |
|---------|-------------|
|Bounded hot key detection | `DebugSnapshot().HotKeys`, `Shards[].HotKeyCandidate` |
| Per-key admission / mitigation | `CheckPerKeyAdmission`, `PerKeyAdmissionSnapshots`, `Mitigations` |
| Shard pressure diagnostics | `PressureSummary()`, `ShardPressure()`, `HotShards()` |
| Autoscaling signals | `Queue.ScaleSignal()` |
| Observability, tests, benchmarks | Hooks, Prometheus metrics, `DebugSnapshot` v0.5 fields |

---

## Reading signals together

```go
snap := q.DebugSnapshot()

// Global coarse depth (admission)
snap.Pressure.IsPressured

// rich diagnostics
snap.PressureSummary.Class
snap.PressureSummary.ScaleRelevant
snap.PressureSummary.MitigationRelevant

// Per shard
for _, sh := range snap.Shards {
    _ = sh.ShardPressure.Class
    _ = sh.HotKeyCandidate
    _ = sh.ShardPressure.HotKeyCandidates
}

// autoscaling signal
snap.ScaleSignal.Recommended
snap.ScaleSignal.Reason
snap.ScaleSignal.Scope
snap.ScaleSignal.DiagnosticsEnabled

// Active per-key mitigation
snap.PerKeyAdmissionSnapshots
snap.Mitigations
```

## Decision flow

```text
1. Is global depth high?        → snap.Pressure, snap.PressureSummary
2. Many shards or one shard?     → snap.PressureSummary.Class
3. Hot key or hot lane?          → ShardPressure.HotKeyCandidates / DominantLane
4. Scale or mitigate?            → ScaleSignal.Recommended / Reason / Scope
                                  → ScaleRelevant vs MitigationRelevant
5. Apply policy                  → overload / lane admission / per-key admission
```

| Pattern | Typical signals | Scale-out helps? | Mitigation helps? |
|---------|-----------------|------------------|-------------------|
| **Localized hot key** | `Reason=localized_hot_key`, low `HotShardCount` | Often **no** | **Yes** — per-key throttle/reject |
| **Hot shard (mixed keys)** | One shard in `HotShards`, no dominant key | Sometimes | Reshard or spread keys |
| **Hot lane** | `lane_dominant`, `DominantLane` set | Sometimes | Lane quota / admission |
| **Distributed backlog** | `distributed`, `many_hot_shards` | **Yes** | Global admission / workers |
| **Worker saturation** | `worker_bound`, high `WorkerBusyRatio` | **Yes** | More workers or faster `Run` |

## When to use which API

| Need | API |
|------|-----|
| Frequent polling (metrics scrape, control loop) | `Queue.ScaleSignal()` |
| Mitigation vs scale classification | `Queue.PressureSummary()` |
| One shard drill-down | `Queue.ShardPressure(shardID)` |
| Full incident bundle | `Queue.DebugSnapshot()` |

Check `DiagnosticsEnabled` on scale and pressure snapshots before interpreting `Reason=none` or empty `Class` — disabled features also return neutral values.

---

## Why CPU and memory can stay flat

Backpressure holds demand in queues instead of executing it. CPU and memory may look healthy while `PressureSummary` or `ScaleSignal` show high depth, wait, reject, or worker-busy ratios.

Use runtime signals — not process CPU alone — to decide when to scale, tune admission, or apply per-key mitigation. See [runbooks/hot-key-and-scale-pressure.md](runbooks/hot-key-and-scale-pressure.md).

## Terminology

Use consistently across docs:

| Term | Meaning |
|------|---------|
| hot key **candidate** | Approximate detection signal — not confirmed root cause |
| localized pressure | One key or shard dominates |
| distributed backlog | Many shards pressured |
| scale **signal** | Advisory metric/API — not an autoscaler |
| mitigation **action** | `allow`, `throttle`, `reject`, `shed` |

Avoid conflating: hot key ≠ hot shard; backpressure ≠ autoscaling; candidate ≠ confirmed root cause.
