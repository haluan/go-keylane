# v0.5 Runtime Signals

Milestone **v0.5.0** adds hot key mitigation, shard pressure diagnostics, and autoscaling signal preparation.

## Task map

| Task | Feature | Primary API |
|------|---------|-------------|
| **KL-1501** | Bounded hot key detection | `DebugSnapshot().Shards[].HotKeyCandidate` |
| **KL-1502** | Per-key admission / mitigation | `CheckPerKeyAdmission`, `PerKeyAdmissionSnapshots` |
| **KL-1503** | Shard pressure diagnostics | `PressureSummary()`, `ShardPressure()`, `HotShards()` |
| **KL-1504** | Autoscaling signals (planned) | `ScaleRelevant`, `MitigationRelevant` on pressure summary |

## Reading signals together

```go
snap := q.DebugSnapshot()

// Global coarse depth (admission)
snap.Pressure.IsPressured

// KL-1503 rich diagnostics
snap.PressureSummary.Class
snap.PressureSummary.ScaleRelevant
snap.PressureSummary.MitigationRelevant

// Per shard
for _, sh := range snap.Shards {
    _ = sh.ShardPressure.Class
    _ = sh.HotKeyCandidate
    _ = sh.ShardPressure.HotKeyCandidates
}

// Active per-key mitigation
snap.PerKeyAdmissionSnapshots
```

## Decision flow

```text
1. Is global depth high?        → snap.Pressure
2. Many shards or one shard?     → snap.PressureSummary.Class
3. Hot key or hot lane?          → ShardPressure.HotKeyCandidates / DominantLane
4. Scale or mitigate?            → ScaleRelevant vs MitigationRelevant
5. Apply policy                  → overload / lane admission / per-key admission
```

## Why CPU can stay flat

Backpressure holds demand in queues. CPU and memory may look healthy while `PressureSummary` shows high depth, wait, or reject ratios. Use runtime signals—not process CPU alone—to decide when to scale or tune admission.
